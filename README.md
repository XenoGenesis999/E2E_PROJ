# E2E_PROJ — End-to-End dbt + Snowflake Data Engineering Project

## Overview

This project implements a full end-to-end data engineering pipeline using **dbt (data build tool)** and **Snowflake**, following the **Medallion Architecture** (Bronze → Silver → Gold). The pipeline ingests raw retail sales data (Superstore dataset), applies progressive data quality transformations, and delivers analytics-ready Gold-layer tables and views for downstream consumption by BI tools and analysts.

The project was built and executed entirely within the **Snowflake dbt Workspace** (Snowsight), with Snowflake serving as the compute and storage engine.

## Data Flow

The pipeline follows the Medallion Architecture across three layers, each applying progressively higher levels of data quality and transformation.

```
PROJ.BRONZE                    PROJ.SILVER                         PROJ.GOLD
───────────────                ───────────────────────             ────────────────────────────
SAMPLE_DATA (source)    ──►    sample_cleaned (View)       ──►    fct_sales (Incremental Table)
                                  Strict cleaning rules           dim_geo   (Table)
                                  ~49 high-integrity rows         dim_prod  (Table)

SAMPLE_DIRTY (source)   ──►    sample_dirty (Table)
                        ──►    sample_clean_relaxed (View)  ──►   segment_region (Table)
                                  Relaxed cleaning rules
                                  ~396 rows
```

### Bronze Layer — Raw Ingestion

| Object | Type | Description |
|--------|------|-------------|
| `SAMPLE_DATA` | Source (external) | Original Superstore CSV data ingested into Snowflake |
| `SAMPLE_DIRTY` | Source (external) | Intentionally corrupted version with nulls, duplicates, invalid timestamps |
| `sample_dirty` | dbt Table | dbt-managed Bronze table built from `SAMPLE_DIRTY`; no transformations applied |

The Bronze layer preserves raw data exactly as ingested. No business logic is applied at this stage. The purpose is to have a reproducible, documented landing zone that dbt can track and test for freshness.

### Silver Layer — Data Cleaning & Quality

| Object | Type | Source | Rows | Rules Applied |
|--------|------|--------|------|---------------|
| `sample_cleaned` | View | `SAMPLE_DATA` | ~49 | Non-null critical fields, valid timestamps (2013–2018), deduplication on `row_id` |
| `sample_clean_relaxed` | View | `SAMPLE_DIRTY` | ~396 | Null handling with COALESCE, relaxed timestamp tolerance, soft deduplication |

Silver views are intentionally materialized as **views** (not tables) to avoid storing intermediate data in Snowflake storage, keeping compute costs minimal while ensuring data is always freshly computed from the Bronze source.

The two Silver paths reflect a deliberate design decision: `sample_cleaned` prioritizes **data integrity** (strict rules, fewer rows) while `sample_clean_relaxed` prioritizes **data volume** (broader tolerance, more rows for trend analysis).

### Gold Layer — Analytics-Ready Outputs

| Object | Type | Source | Description |
|--------|------|--------|-------------|
| `fct_sales` | Incremental Table | `sample_cleaned` | Central fact table; one row per sales transaction; keyed by `row_id` |
| `dim_geo` | Table | `sample_cleaned` | Geography dimension; deduplicated by `geo_id` (SHA-256 hash of country\|\|state\|\|city\|\|region) |
| `dim_prod` | Table | `sample_cleaned` | Product dimension; deduplicated by `product_id` (SHA-256 hash of category\|\|sub_category) |
| `segment_region` | Table | `sample_clean_relaxed` | Aggregated mart; `total_sales`, `total_profit`, `total_quantity` by segment + region + date |

Gold models are materialized as **tables** to ensure fast query performance for BI tools. `fct_sales` uses an **incremental materialization** strategy (`delete+insert`) to support efficient loading in a production scenario where new rows are appended over time.

## Commands Used

All commands were executed within the Snowflake dbt Workspace environment targeting the `dev` profile.

```bash
# Install dbt package dependencies
dbt deps

# Compile the project (validate Jinja, check for syntax errors, build the DAG)
dbt parse

# Run all models (build Bronze, Silver, Gold in dependency order)
dbt run

# Run models selectively (e.g., only Gold layer)
dbt run --select fct_sales dim_geo dim_prod segment_region

# Run all tests (unique, not_null, accepted_values, relationships, custom)
dbt test

# Run tests for specific models
dbt test --select fct_sales dim_geo dim_prod

# Generate documentation artifacts (produces catalog.json + index.html in target/)
dbt docs generate

# List all project nodes (verify DAG structure)
dbt ls
```

## Data Quality Tests

dbt tests were defined in YAML files alongside each Gold model. All tests passed successfully (confirmed by green "Success" badges in the workspace DAG).

### Built-in Tests

| Model | Column | Test Type | Rule |
|-------|--------|-----------|------|
| `fct_sales` | `row_id` | `unique` | No duplicate transaction keys |
| `fct_sales` | `row_id` | `not_null` | Every row must have a key |
| `fct_sales` | `segment` | `accepted_values` | Must be one of: CONSUMER, CORPORATE, HOME OFFICE |
| `fct_sales` | `region` | `accepted_values` | Must be one of: EAST, WEST, CENTRAL, SOUTH |
| `fct_sales` | `ship_mode` | `accepted_values` | Must be one of: STANDARD CLASS, SECOND CLASS, FIRST CLASS, SAME DAY |
| `fct_sales` | `region` | `relationships` | Every region must exist in `dim_geo` |
| `fct_sales` | `category` | `relationships` | Every category must exist in `dim_prod` |
| `dim_geo` | `geo_id` | `unique` + `not_null` | No duplicate or null geography keys |
| `dim_prod` | `product_id` | `unique` + `not_null` | No duplicate or null product keys |
| `dim_prod` | `category` | `accepted_values` | Must be: FURNITURE, OFFICE SUPPLIES, TECHNOLOGY |

### Custom Generic Test

A custom test `valid_order_ts` was implemented in `tests/generic/valid_order_ts.sql` to enforce a domain-specific business rule:

```sql
{% test valid_order_ts(model, column_name) %}
  select {{ column_name }}
  from {{ model }}
  where {{ column_name }} < '2013-01-01'::timestamp
     or {{ column_name }} > '2018-01-01'::timestamp
{% endtest %}
```

**Rule**: All `order_ts` values must fall within the known Superstore dataset period (January 2013 – January 2018). Any timestamp outside this range is flagged as a data quality violation. This test is applied to `fct_sales.order_ts` and demonstrates dbt's extensibility beyond built-in tests.

## Assumptions

The following assumptions were made during the design and implementation of this project:

1. **Static dataset**: The Superstore dataset is treated as a static historical dataset (2013–2018). The incremental strategy on `fct_sales` is implemented as a production best practice pattern, even though the source data does not change in this demo.

2. **Two Silver paths are intentional**: `sample_cleaned` (strict) and `sample_clean_relaxed` (relaxed) represent different analytical use cases — the strict path powers dimensional modeling, the relaxed path powers broader volume-based aggregation.

3. **Snowflake views for Silver**: Silver models are intentionally materialized as views rather than tables to minimize storage costs. In a production setting with large data volumes, these could be converted to incremental tables.

4. **SHA-256 hashing for dimension keys**: Surrogate keys in `dim_geo` and `dim_prod` are generated using `SHA2(CONCAT_WS('||', ...))` rather than auto-increment integers, making them portable across environments and safe for incremental loads.

5. **Direct Snowflake view reference in Gold**: Gold models reference `PROJ.SILVER.SAMPLE_CLEANED` directly (without a dbt Silver model `ref()`) for `fct_sales`, `dim_geo`, and `dim_prod`. This is a valid pattern when the Silver object is a pre-existing Snowflake view not managed within the same dbt project.

6. **Timestamp validity window**: The range `2013-01-01` to `2018-01-01` was chosen based on the known date range of the Superstore dataset. Any records outside this range are considered data quality violations.

## Challenges Faced

### 1. Custom test placement error
**Problem**: The custom test `valid_order_ts.sql` was initially placed at `tests/valid_order_ts.sql`. dbt treats files at the root of `tests/` as **singular tests** (standalone SQL), not reusable generic test macros. This caused a compilation error:
> `'test_valid_order_ts' is undefined`

**Resolution**: Moved the file to `tests/generic/valid_order_ts.sql`. dbt scans the `tests/generic/` directory specifically for reusable generic test macros, resolving the reference error.

### 2. `ref()` resolution failure for Silver models
**Problem**: Gold models initially used `{{ ref('SAMPLE_CLEANED') }}`, which caused a compilation error:
> `Model depends on a node named 'SAMPLE_CLEANED' which was not found`

`ref()` in dbt only resolves to dbt-managed model files — it cannot reference pre-existing Snowflake views that were created outside dbt.

**Resolution**: Replaced `{{ ref('SAMPLE_CLEANED') }}` with a direct Snowflake fully qualified reference: `PROJ.SILVER.SAMPLE_CLEANED`. This bypasses dbt's dependency graph for that reference but allows the models to build correctly.

### 3. `dbt docs generate` not running automatically
**Problem**: The Snowflake workspace runs `dbt run` and `dbt test` via job configuration, but `dbt docs generate` requires a separate explicit invocation to produce `catalog.json` and `index.html` in the `target/` directory.

**Resolution**: Added `dbt docs generate` as an explicit step in the workspace job configuration. Confirmed success by verifying `catalog.json` appeared in `target/`.

### 4. Determining row counts across Silver paths
**Problem**: It was not immediately obvious how many rows would survive the strict vs. relaxed cleaning rules without running queries against the Silver views.

**Resolution**: Executed `SELECT COUNT(*) FROM PROJ.SILVER.SAMPLE_CLEANED` and `PROJ.SILVER.SAMPLE_CLEAN_RELAXED` directly in Snowsight to confirm 49 and 396 rows respectively, validating that the cleaning logic behaved as intended.

## Future Improvements

### Data Pipeline

- **Add Snowpipe or COPY INTO automation**: Automate the Bronze ingestion step so new CSV files dropped into a Snowflake stage are automatically loaded, making the pipeline truly end-to-end.
- **Implement SCD Type 2 snapshots**: Use dbt's `snapshot` functionality to track slowly changing dimensions (e.g., customer segment changes over time) in `dim_geo` and `dim_prod`.
- **Convert Silver views to incremental tables**: For large-scale production data, Silver views should be materialized as incremental tables to avoid full recomputation on every Gold model query.

### Data Quality

- **Add source freshness checks**: Define `freshness` blocks in `src_superstore.yml` to alert when Bronze source data has not been updated within an expected SLA window.
- **Expand custom tests**: Add additional custom generic tests for business rules such as `sales > 0`, `discount BETWEEN 0 AND 1`, and `profit > -sales` (profit cannot exceed full loss).
- **Add dbt expectations package**: Integrate `dbt-expectations` (Great Expectations wrapper) for richer statistical tests such as column value distributions, row count ranges, and regex pattern matching.

### Documentation & Governance

- **Add column-level descriptions**: Populate `description:` fields for every column in all YAML files to enrich the dbt docs catalog and make the project self-documenting.
- **Add exposures**: Define `exposures:` in dbt to document which BI dashboards or downstream tools consume which Gold models, completing the full lineage from source to consumption.
- **Integrate with dbt Cloud**: Migrate from the Snowflake workspace to dbt Cloud for scheduled job runs, CI/CD integration (pull request checks), and hosted dbt docs with team-level access.

### Performance & Scalability

- **Cluster Gold tables**: Add `cluster_by` on `fct_sales` (e.g., by `order_date` or `region`) to improve Snowflake's micro-partition pruning for time-series and regional queries.
- **Add result caching policy**: Configure Snowflake's query result cache and warehouse auto-suspend/auto-resume settings to minimize compute costs for repeated BI queries on Gold tables.
- **Parameterize environments**: Extend `dbt_project.yml` to support multiple targets (`dev`, `staging`, `prod`) with different databases and warehouses, enabling a proper CI/CD promotion workflow.
